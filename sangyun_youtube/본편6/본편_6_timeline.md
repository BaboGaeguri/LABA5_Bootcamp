## [00:30 – 03:00] VLA란 무엇인가

VLA는 어떤 정보던 토큰화를 할 수 있다면 모델이 그 맥락을 파악해서 다음에 올 토큰을 예측할 수 있다는 아이디어를 실제 행동(Action)에 확장한 모델입니다.

기존의 로봇 정책들은 특정 작업에 특화되서 행동을 수행하는데에는 익숙했지만, 새로운 환경과 학습되지 않은 작업에서는 전혀 그 성능을 내지 못했습니다.

VLA는 이런 문제점을 해결하여 범용 로봇을 지향하는 정책이라고 이해할 수 있겠습니다.

## [06:30 – 12:00] 환경 구성

개인적으로 이번 편에서는 제가 so-arm을 하이리온의 팔로 만들기 위한 과정을 좀 더 자세히 설명해보고 싶습니다. 영상을 보시는 분들이 영상을 보고 더 좋은 방법이 있었거나, 잘못된 방법으로 진행한 점이 있다면 알려주시면 감사하겠습니다.

자막: 진짜 궁금해서 그럼

이전에 말씀드린 것처럼 제가 비전공자이기도 하고 프로젝트 기간도 제한이 있기 때문에 모든 것을 기초 단위에서 이해하고 진행하는 것은 불가능하다는 생각이 들었습니다. 그래서 AI가 제가 원하는 결과물을 내놓을 수 있도록 최대한 환경을 잘 구성하는 데 집중했던 것 같습니다.

일단 학습 토르소를 최종 하이리온과 동일한 높이로 제작하여 최대한 학습 환경과 실제 운영 환경을 일치시키려고 했습니다. 리더 암에 대한 토르소도 따로 제작을 했습니다.

so-arm의 학습을 위해서 총 5대의 컴퓨터를 사용했습니다.

자막: 이렇게 보니 정말 난잡하군요..

각 기기의 역할은 아래와 같습니다.

| 기기 | 역할 |
|------|------|
| 내 노트북 | dev-PC |
| DGX | Data collector |
| 교수님 PC | Trainer |
| Jetson | Inference |
| 성래 노트북 | Data collector (미사용) |

Jetson과 DGX는 dev-PC와 SSH 연결을 통해서 dev-PC의 AI가 나머지 두 컴퓨터의 환경을 인지하고서 작성된 코드들을 다시 두 컴퓨터로 배포하도록 했습니다.

RTX 3090이 달린 교수님 컴퓨터는 SSH 연결을 하기에는 번거로웠기 때문에 GitHub를 통해 코드를 공유했습니다.

결과적으로 DGX에서 데이터를 수집해서 HuggingFace에 저장하고, 교수님 PC에서 데이터를 받아 학습하여 HuggingFace에 저장하고, Jetson에서 모델을 받아 추론하고, dev-PC는 이 과정을 총괄하는 구조로 작업을 했습니다.

그리고 에이전트 환경 구성은 Era / Spec / Todo 문서 구조를 기반으로, 각 기기별 역할에 맞는 에이전트를 정의하고 smolVLA 서브레포를 레퍼런스로 연결하는 방식으로 잡았습니다.

## [12:00 – 16:00] 학습 과정

이렇게 구성된 환경 안에서 어떻게 모델을 학습시켰는지에 대해서 말을 해보겠습니다.

일단 저희가 사용한 smolVLA는 크게 VLM 백본과 Expert layer 두 가지 파트로 나뉩니다. VLM을 같이 학습시킬 건지, expert만 학습시킬지 여부와 함께 LoRA를 적용할지와 Full FT를 기준으로 6개의 축으로 나눈 후, 데이터의 양이 얼마나 되느냐에 따라 원하는 성능에 도달하나와 학습 효율은 어떻게 되는가를 핵심 고려사항으로 하여 6개의 학습 방법 중 하나를 선택하는 흐름으로 진행하고자 했습니다.

크게 base, ep40, ep100, ep310개로 나눠볼 수 있었던 것 같습니다.

하이리온의 팔 배치와 카메라 세팅을 기준으로 base를 돌려봤을 때 전혀 움직이지를 못해서 expert 레이어만 학습시키면 안되고 VLM까지 같이 건드려야 함을 알 수 있었습니다.

ep40개와 100개를 각각 해봤을 때는 물체를 더 잘 인식하는 듯 보였으나 여전히 제대로 물체를 잡지는 못해서 데이터를 더 모으거나 학습 방법을 바꿔야 한다는 두 가지 가설이 있었습니다. 일단 데이터를 300개까지 모으고 학습을 해 봤을 때도 문제가 있다면 학습 방법을 다르게 가져가기로 하고 데이터를 더 늘렸습니다. 그러면서 기본 축을 건들지는 않으면서 학습의 최적화를 위한 파라미터들도 조금씩 건드렸었습니다.

## [16:00 – 20:00] 안됐던 이슈들

진행 과정에서 이슈도 여러 개가 있었습니다. 제가 기억하는 가장 큰 이슈는 두 개입니다.

일단 첫째로 DGX를 왜 데이터 수집기로만 사용하는가에 대한 문제입니다. 저도 DGX의 강력한 성능으로 학습을 하려고 했으나 결국 RTX 3090이 장착된 교수님 PC를 사용할 수밖에 없었습니다. 그 이유는 video decode backend default가 torchcodec인데, 현재 DGX에서는 ABI 미스매치가 발생하여, fallback으로 pyav가 작동하나 메모리 leak이 발생해 OOM이 떴기 때문입니다.

또한 Orin에서도 문제가 있었는데요. LeRobot이 공식적으로 지원하는 Python 버전이 3.12+인데, 현재 Orin JetPack 기준으로는 3.12+가 지원되는 PyTorch가 존재하지 않습니다. 그래서 Python 환경을 3.10으로 맞춘 후 LeRobot 소스코드를 이 문법에 맞게 수정하게 되었습니다.

이를 통해 다양한 기기들 간의 호환성이 정말 중요한 것이구나와, 이를 맞춰줘서 내가 원하는 하드웨어에서 원하는 소프트웨어를 돌아가게 하는 것이 생각보다 쉽지 않은 길이라는 것을 느낄 수가 있었습니다.

## [22:00 – 25:00] 모델의 한계

험난하긴 했지만 이런 크고 작은 문제들을 해결할 건 해결하고, 우회할 건 우회해서 학습을 완료했습니다. 조명도 줄여 보고, 같은 공간이지만 학습할 때 봤던 view와 다른 공간에서 추론을 시켜봤을 때도 물체를 잘 잡았습니다.

문제는 시연장에서 돌려보니, 조명과 배경 등이 학습 환경과 많이 달라져서 그런지 애가 정신을 못 차리는 현상을 발견할 수 있었습니다.

결과적으로 각 task별로 150개씩의 에피소드로 학습을 했음에도 범용적인 환경에서 작동하는데는 어려움이 있는 것을 확인할 수 있었습니다.

저는 개인적으로 이 문제를 모델의 한계라고 조심스럽게 추정하고 있는데요. overview 카메라를 원래의 so-arm과 다르게 배치했기 때문에 VLM 백본의 학습 데이터 분포와 달라졌기 때문에 그랬을 가능성이 있을 수 있을 것 같고, smolVLA 자체가 경량화된 VLA 모델이다 보니 상대적으로 성능이 저조할 수도 있겠다는 생각도 들었습니다.

결국에는 진짜로 범용적인 모델이라고 할 수 있을 정도라면 엄청나게 많은 양과 관점에서 생산된 데이터를 때려넣어야 가능한 것이지 않을까라는 생각이 들었습니다. 물론 범용적이라는 것의 정의를 어떻게 하느냐에 따라 달라질 수도 있는 문제이긴 하지만, 범용 로봇이라는 건 기술적인 문제를 넘어서 자본적인 문제가 필히 수반된다는 점을 알 수가 있었습니다. Open AI가 아직까지도 적자를 못 벗어나고 있는데, 피지컬에서는 이보다 더하면 더했지 덜 한 문제가 있지는 않을 것 같아 보였습니다.
